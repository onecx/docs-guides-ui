== Microfrontend Loading in OneCX

This guide helps you understand the sequence of events that occur when a OneCX application loads a microfrontend.

=== Application Loading Phases

*Application loading* can be divided into 5 phases:

[mermaid]
----
graph LR
  preload[Preload] --> bootstrap[Bootstrap]
  bootstrap --> config[Configure]
  config --> route[Route]
  route --> remote[Load Remote]
----

=== Execution Flow
Goal indicates the purpose of each phase, steps describe the key actions taken, and verify lists ways to confirm that the phase completed successfully.

.Step 1: Preload Phase
*Goal* - Prepare the Module Federation runtime and register shared scopes before the shell boots

*Steps*

* Load the preloader bundles that register shared scopes and Module Federation runtime helpers
* Wait for each preloader to confirm that the runtime is ready
* Only then import and bootstrap the Angular application

*Verify*

* *Network*: Preloader remoteEntry scripts are fetched successfully
* *Console*: Logs show preloader registration and runtime-ready messages
* *Runtime*: window.onecxPreloaders contains registration info

.Step 2: Bootstrap Phase

*Goal* - Boot the Angular application and run app initializers that load runtime configuration

*Steps*

* The Angular application is bootstrapped
* APP_INITIALIZER providers run during bootstrap and do the following:
    - fetch workspace configuration
    - populate configuration with runtime properties (APP_BASE_HREF, endpoints, flags)
    - initialize App state, permissions, and global services
* After APP_INITIALIZER completes, the router and application UI become interactive

*Verify*

* *Console*: App module bootstrapped log is present
* *Network*: Workspace configuration loaded successfully
* *Runtime*: ConfigurationService contains expected properties (APP_BASE_HREF, routes)
* *Router*: Initial routes are registered

.Step 3: Configure Phase

*Goal* - Load and validate runtime workspace configuration used to build routes and remote URLs

*Steps*

* Fetch workspace configuration that contains metadata, route definitions, remoteEntry URLs, permissions, and feature flags
* Store runtime values in configuration, such as APP_BASE_HREF, endpoints, and feature flags
* Validate and normalize remoteEntry URLs and exposedModule names
* Prepare permissions and feature flags used by RoutesService and guards

*Verify*

* *Network*: BFF response contains route entries
* *Runtime*: Check if configuration service returns expected values
* *Logs*: Config fetch succeeded and entries are validated

.Step 4: Route Phase

*Goal* - Convert workspace routes into Angular Route objects with lazy loaders

*Steps*

* On route initialization, convert workspace routes to Angular Route objects
* Each route uses loadChildren that calls the Module Federation runtime loader to load the remote module on demand
* Attach guards, permissions, and titles as provided by the workspace config
* Register default and static and dynamic routes

*Verify*  

* *Console*: Route creation logs are present showing the path
* *Navigation*: Navigating to a registered path triggers loadChildren

.Step 5: Remote Loading Phase

*Goal* - On route activation, fetch the remoteEntry, resolve the exposed module, prepare the shell, and mount the microfrontend

*Steps*

* On navigation, call loadRemoteModule with the remoteEntry URL and exposed module name for the active route
* loadRemoteModule fetches remoteEntry.js, initializes sharing, and resolves the exposed module
* If the technology is Angular, return the module; otherwise, return WebcomponentLoaderModule
* Update styles/themes, fetch remote permissions, update AppStateService, and show loading state
* Load the route linked module into the router outlet

*Verify*

* *Network*: remoteEntry.js is fetched successfully
* *Console*: Module Federation init and remote load logs are present
* *UI*: Remote content renders inside
