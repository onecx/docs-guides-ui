= Use of Module Federation in Shell router

*Shell Router* is the center for dynamic micro frontend loading. *Module federation* is used in *Shell Router* to:

    * Configure routes to use *Module federation*
    * Loads application modules via *Module federation*

== BFF Routes to Angular Routes Transformation

The workspace configuration contains BFF (Backend For Frontend) route definitions. The Shell Router transforms these backend route configurations into Angular application routes:

* Receives route configuration from *workspace configuration* (backend definition)
* Converts each BFF route into an Angular lazy-loaded route
* Configures the loading mechanism using *Module Federation* and path resolution
* Merges routes with static Shell routes

== Shell Routes After Transformation

After transformation, the Shell Router contains two types of routes:

=== Static Routes

Shell has following static routes:

* Error pages
* About Shell page

These routes are merged with application routes to create final list of application routes

=== Application Routes

Dynamically configured routes based on workspace configuration:

* Each application route is a lazy-loaded route
* Route activation triggers *Module Federation* to load the remote micro frontend
* The route definition includes:
    - Path matcher
    - Module Federation configuration - *remote name and export path*
    - Micro frontend type - *WebComponent or Angular module*
    - Loading mechanism callback

== How *Shell Router* works with *Module Federation*?

Following steps are performed in *Shell Router* to achieve dynamic loading of micro frontends using *Module Federation*:

=== Router Configuration Phase

*Shell router* converts router configuration received from *workspace configuration* into *application route configuration*:

* Parse workspace router configuration
* For each route, configure:
    - Path and routing rules
    - MFE module federation details (remote, exposed module)
    - Loading mechanism (WebComponent or Angular module)
    - Callback to handle dynamic loading via *Module federation*
* Merge static Shell routes with application routes

=== Runtime Loading Phase

When a route is activated, dynamic loading of micro frontend is triggered:

* *Route activation* triggers lazy loading of the configured MFE
* *Module Federation* loads the remote micro frontend
* Module federation supports *script* and *module* type of loading. Detailed information can be found here- https://www.angulararchitects.io/en/blog/dynamic-module-federation-with-angular/[Dynamic module federation with angular]
* *Negotiate versions* based on current state of federation instance and shared dependencies
* *Decide MFE rendering method* based on MFE type configuration (see xref:./types-of-microfrontends.adoc[Types of Microfrontends])
* *Render* the loaded MFE using the appropriate method

== Rendering methods

[.stripes-even,cols="20,~"]
|===
| *Method* | *Implementation Details*

| WebComponent 
a|
*How it works:* Display the MFE as a designated HTML custom element tag

*Steps:*

. Shell Router identifies the custom element tag name from MFE configuration
. Loads the MFE module via Module Federation
. Appends the custom element tag to a loader component
. The web component manages its own initialization, state and lifecycle

| Angular Module 
a|
*How it works:* Lazy load and bootstrap the exposed Angular module using module federation

*Steps:*

. Shell Router loads the MFE via Module Federation
. Retrieves the exposed Angular module from the remote entry
. Bootstraps the module within the Shell's Angular injector context
. Renders the module's root component inside the Shell Router outlet
|===
